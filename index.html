Python Security Best Practices

The most common Python security vulnerabilities and how to fix them.

Dependency security

Pin all dependencies to exact versions (==) in production. Unpinned deps (requests>=2.0) can auto-upgrade to vulnerable versions. Run pip-audit or this scanner in CI before every deployment. Update dependencies regularly — most CVEs are fixed within days of disclosure. Use pip install --require-hashes to prevent supply chain attacks.

Never use eval() or exec()

eval(user_input) is arbitrary code execution — any string the attacker controls becomes Python code running with your process permissions. Replace with: JSON parsing for data (json.loads()), ast.literal_eval() for safe Python literal evaluation, or explicit parsing logic. There is no safe way to use eval() with untrusted input.

Pickle deserialization

pickle.loads(data) on untrusted data is equivalent to eval() — it can execute arbitrary Python code during deserialization. An attacker who controls the pickled bytes can run any command. Use JSON, MessagePack, or Protocol Buffers for data exchange with untrusted sources. Only unpickle data you serialized yourself from a trusted source.

Weak cryptography

MD5 and SHA1 are broken for security purposes — never use for password hashing or integrity verification. For passwords: use bcrypt, argon2-cffi, or hashlib.scrypt(). For integrity: use SHA-256 minimum. For random tokens and session IDs: use secrets.token_hex() not random.random() — the random module is not cryptographically secure.

Hardcoded Secrets — The Most Common Mistake

Hardcoded API keys, database passwords, and tokens in source code are the leading cause of cloud breaches. Git history is permanent — even if you remove a secret in a later commit, it remains in the history. Use environment variables (os.environ.get('SECRET_KEY')), python-dotenv for local development, and a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) in production. Scan git history with truffleHog or gitleaks if you suspect a commit contained a secret.

Python Security Tools – FAQ

How does the dependency vulnerability scanner work?+
The scanner parses your requirements.txt to extract package names and version numbers. For each package, it sends a POST request to the OSV.dev API (osv.dev/v1/query) with the package name and version. OSV returns any known vulnerabilities from the PyPA Advisory Database, GitHub Security Advisories, and other sources. Results include the CVE/GHSA ID, severity (CRITICAL/HIGH/MEDIUM/LOW), a description of the vulnerability, and the version that fixes the issue. All data comes directly from OSV.dev — the same database used by pip-audit and GitHub Dependabot.
Is my code or requirements.txt sent to any server?+
The code scanner is 100% client-side — your Python code never leaves your browser. For the dependency scanner, only package names and version numbers are sent to the OSV.dev API (no code, no file contents). OSV.dev is a Google-maintained public database — it receives billions of queries from tools like pip-audit and Dependabot. The OSV.dev privacy policy applies to those queries. If you're scanning code from a confidential project, use the code scanner (client-side) freely, but be aware the dependency scanner sends package metadata to osv.dev.
What is the difference between pip-audit and this tool?+
pip-audit is a CLI tool you install with pip that scans your current environment or a requirements file. This tool does the same dependency scanning through a browser UI — no installation needed. pip-audit is better for CI/CD pipelines, automated scanning, and scanning virtual environments. This tool is better for quick one-off checks without installing anything. Both use the OSV.dev database. pip-audit also checks against PyPA's own advisory database which may have slightly different coverage than OSV alone.
What Python security patterns does the code scanner detect?+
The code scanner detects: eval() and exec() usage (arbitrary code execution risk), pickle.loads/dumps/Unpickler (unsafe deserialization), subprocess with shell=True (command injection), os.system() calls, hardcoded passwords/secrets/API keys, MD5 and SHA1 for cryptography (weak hashing), random.random() in security contexts (non-cryptographic PRNG), SQL string concatenation (SQL injection risk), assert statements for security checks (bypassable with -O), yaml.load() without Loader (unsafe YAML), urllib without SSL verification, and more. It uses pattern matching — not AST analysis like Bandit — so review all findings in context.
How do I run Python security scanning in CI/CD?+
For CI/CD, use CLI tools: pip-audit (pip install pip-audit, then pip-audit -r requirements.txt --fail-on-vuln) for dependency scanning, and Bandit (pip install bandit, then bandit -r src/) for code scanning. Both return non-zero exit codes when vulnerabilities are found, which fails the CI build. For GitHub Actions: use the official pip-audit GitHub Action or add the commands as a workflow step. Safety CLI also integrates with CI/CD and has a GitHub Action. This browser tool is for quick checks — for production pipelines, use the CLI tools directly.
